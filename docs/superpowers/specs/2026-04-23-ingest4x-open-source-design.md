# ingest4x 开源化重构设计

## 背景

当前仓库来源于一个自用收数服务，已经具备完整的服务形态：

- HTTP 接入
- rules 加载与校验
- mock / local 运行模式
- Kafka / Redis 生产依赖
- JLT 校验工具链

但从开源产品角度看，仓库仍然保留了明显的内部业务命名和历史边界：

- `up` / `ad` 是内部语义，不适合作为开源产品的主叙事
- `attribution_status` / `caid` 这类能力强依赖特定业务背景与 SDK 协同
- 路由、文档、配置和代码命名还没有围绕统一产品概念收敛

本次重构的目标不是把项目做成框架，而是把它整理成一个可以直接部署、可以本地端到端跑通的开源收数服务。

## 目标

- 将项目对外统一定义为 `ingest4x`
- 将对外主能力统一定义为 `ingest`
- 提供一套可直接部署的通用收数服务叙事
- 保留当前请求字段模型，不改 `xwhat` / `xcontext` 等核心协议
- 默认支持本地 `mock/local` 跑通
- 保留 Kafka / Redis 作为生产模式依赖
- 新增 `ingest` 新入口，同时保留旧 `up` / `ad` 作为迁移参考
- 为未来 sink / lookup 可插拔预留边界，但本次不做框架化改造

## 非目标

- 不将项目重构为可插拔框架或插件系统
- 不在本次改动中统一响应体或 HTTP 状态码语义
- 不修改现有请求字段命名与载荷结构
- 不将 `up` / `ad` 立即删除
- 不保留 `attribution_status` 和 `caid` 进入开源版 v1
- 不处理 SDK 配套、归因生态或特定客户功能抽象

## 产品定位

`ingest4x` 的开源版 v1 被定义为：

“一个可直接部署、默认可本地端到端跑通的通用收数服务。”

它优先解决的问题是：

- 本地可以零外部依赖体验服务能力
- 生产环境可以接入 Kafka / Redis 运行
- 用户可以通过 rules 和配置完成收数行为的配置
- 用户可以通过内置工具链验证规则和示例

它不优先解决的问题是：

- 为外部开发者提供一套框架式 API
- 支持复杂扩展生命周期
- 抽象出通用插件发现与加载机制

## 能力范围

开源版 v1 正式支持范围为：

- `POST` 方式的 JSON 收数
- `GET` 方式的 querystring 收数
- rules 校验
- event normalization
- mock/local 运行
- Kafka / Redis 生产运行
- JLT 验证能力

明确排除的范围为：

- `attribution_status`
- `caid`
- 广告跳转、短链落地页跳转等非核心 ingest 行为

## 接口设计

### 对外主接口

开源版新增统一入口：

- `POST /ingest`
- `GET /ingest`

其中：

- `POST /ingest`
  - 承接当前 `up` 风格能力
  - 输入来自 JSON body
- `GET /ingest`
  - 承接精简后的 `ad` 风格能力
  - 输入来自 path / query / headers 中与收数相关的参数

### 兼容性约束

以下内容在 v1 明确保持不变：

- 请求字段结构
- `xwhat` / `xcontext` 等现有核心字段
- 各类事件字段语义
- 成功或失败时的既有返回值行为

这意味着 v1 的变化重点是：

- 产品概念统一
- 路由统一
- 模块边界整理
- 开源文档和本地体验整理

而不是一次性修改接入协议。

### 旧接口策略

以下旧入口在迁移期继续保留：

- `/up`
- 现有 `ad` 路由

保留目的：

- 作为 `/ingest` 新接口的行为参考
- 用于回归验证和差异比对
- 降低重构过程中的协议回归风险

迁移期内，旧接口不再作为新能力主入口继续扩展。

## 运行模式设计

### 本地模式

本地模式是开源版一等能力，而不是测试残留物。

要求：

- 默认能本地跑通
- 不依赖 Kafka
- 不依赖 Redis
- 可以通过 mock lookup / mock sink 完成端到端验证
- README 第一条路径必须是本地快速启动

### 生产模式

生产模式继续使用：

- Kafka 作为事件投递主通道
- Redis 作为必要 lookup 依赖

开源版文档需要明确区分：

- 本地体验路径
- 生产部署路径

避免让初次使用者一上来就陷入复杂外部依赖。

## 模块边界设计

本次重构不做完整框架化，但需要把代码结构整理到能支撑未来演进的状态。

建议边界如下：

- `cli`
  - 承载 `server`、`test`、版本输出等命令行入口
- `app`
  - 承载服务启动、路由注册、应用状态装配
- `ingest`
  - 承载新的统一 ingest 入口与核心流程
- `legacy`
  - 暂存旧 `up` / `ad` 逻辑，作为迁移参考
- `rules`
  - 承载规则加载、合并、校验
- `infra`
  - 承载 Kafka、Redis、mock、cache、logging 等运行时能力
- `settings`
  - 承载配置模型与运行模式定义

目标不是为了抽象而抽象，而是让新 `ingest` 入口能独立生长，同时不影响旧逻辑对照和回归。

## 新旧能力关系

### 新能力

`ingest` 是开源版的正式入口层。

内部可以先拆成两类输入适配：

- JSON 输入路径
- querystring 输入路径

两者最终都进入统一的 ingest 处理链路。

### 旧能力

`up` / `ad` 暂时保留，但角色发生变化：

- 作为参考实现
- 作为行为对照物
- 作为迁移期的回归基线

它们不再承担开源版产品命名与对外主叙事。

## 迁移顺序

本次重构应采用渐进式迁移，而不是大爆炸式替换。

### 阶段 1：建立 `ingest4x` 对外骨架

- 调整 README 主叙事
- 调整产品命名为 `ingest4x`
- 明确开源版 v1 范围
- 保留旧代码结构不动或少动

### 阶段 2：新增 `POST /ingest`

- 先对齐当前 `up` 行为
- 确保请求字段兼容
- 确保 mock/local 可运行
- 用旧 `/up` 作为行为对照

### 阶段 3：新增 `GET /ingest`

- 对齐精简后的 `ad` 收数行为
- 删除跳转相关逻辑
- 将 querystring 接入能力收敛到 ingest 概念

### 阶段 4：抽公共能力

在新接口已经可用后，再逐步抽取共享能力，例如：

- 规则校验
- lookup
- normalization
- sink 投递
- 响应映射

### 阶段 5：旧接口降级

- 文档中将 `/up` 和旧 `ad` 标记为 legacy/reference
- 新增能力只落在 `/ingest`
- 旧入口只保留回归和兼容角色

### 阶段 6：开源化收尾

- 清理客户强绑定内容
- 补齐示例配置
- 补齐本地运行文档
- 补齐发布与测试说明

## 文档策略

开源版文档必须围绕 `ingest4x` 展开，而不是让用户先学习内部历史命名。

README 应明确提供：

- 这是什么
- 适合什么场景
- 5 分钟本地跑通方式
- mock/local 与 production 的区别
- `POST /ingest` 与 `GET /ingest` 的使用方式
- rules / JLT 的最短学习路径

文档不应再以 `up` / `ad` 作为首页核心目录，而应把它们降为迁移期术语或历史参考。

## 测试策略

本次重构需要同时覆盖新入口和旧入口对照。

至少需要验证：

- `POST /ingest` 在 mock/local 模式可工作
- `GET /ingest` 在 mock/local 模式可工作
- 关键请求字段与旧链路行为一致
- 旧 `/up` 与新 `POST /ingest` 在同类输入上结果可比对
- 精简后的 `GET /ingest` 不再依赖跳转相关行为
- 规则与 JLT 工具链不因开源化命名收敛而失效

不应把“已有旧测试通过”误认为“新入口已稳定”，因为新入口本身是新的产品承诺，需要独立验证。

## 风险与控制

### 风险 1：产品命名变了，但代码仍然是旧语义拼接

如果只改文档和路由，不整理模块边界，开源后很快会出现“表面叫 ingest，内部还是 up/ad 拼装”的理解割裂。

控制方式：

- 新入口单独建层
- 旧逻辑明确标注为 legacy/reference

### 风险 2：一次改太多，导致协议兼容回归

如果同时改命名、路由、字段、返回值和运行模式，重构风险会急剧上升。

控制方式：

- 锁定请求字段不变
- 锁定返回值不变
- 先做新入口，再做共享逻辑抽取

### 风险 3：本地体验仍然依赖生产组件

如果 mock/local 只是“理论支持”，但文档和默认路径仍要求 Kafka / Redis，开源产品的第一印象会很差。

控制方式：

- 本地模式作为正式一等场景
- 示例配置优先以本地运行展示

### 风险 4：旧能力删除过早

如果在没有建立足够对照测试之前就删除旧 `/up` / `ad`，将难以判断新逻辑是否真的兼容。

控制方式：

- 旧入口先保留
- 待新入口稳定后再考虑下线计划

## 决策结论

本次开源化重构采用以下决策：

- 项目对外命名统一为 `ingest4x`
- 新增统一对外接口 `ingest`
- `POST /ingest` 对应当前 JSON 收数链路
- `GET /ingest` 对应当前 querystring 收数链路
- `up` / `ad` 暂时保留为迁移参考，不再作为主叙事
- `attribution_status` / `caid` 从开源版 v1 中移除
- 请求字段和返回值保持现状
- 默认提供 mock/local 端到端运行体验
- 生产模式继续基于 Kafka / Redis
- 内部结构为未来可插拔演进预留边界，但本次不框架化

后续实现应主要围绕以下范围展开：

- `src/main.rs`
- `src/lib.rs`
- `src/server.rs`
- `src/handlers/`
- 新增 `src/ingest/` 或等价新模块
- `src/settings.rs`
- `src/utils/`
- `README.md`
- `docs/features/*`
- `docs/recommended-workflow.md`
- `examples/`
- 对应测试文件

实现阶段应继续遵守以下约束：

- 先新增 `ingest`，不先删旧接口
- 先保证本地 mock/local 跑通
- 先做行为对齐，再做抽象清理
